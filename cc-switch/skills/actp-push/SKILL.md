# actp-push · ACTP 团队推送前审查 + 推送

按 ACTP 项目组规范执行"推送前同步 + 分支对齐 + 变更摘要 + 冲突诊断 + 推送"完整流程。**核心价值**: 不是简单 `git push`, 而是在推送前主动发现本地与远程的差异, 解释冲突原因, 避免覆盖别人代码。

## 适用角色

- 普通组员(推送自己的个人分支)
- 小组负责人(推送负责人分支,前提是已经合并完所有组员)
- yra_dev 总负责人(推送 yra_dev,前提是已经合并完所有负责人分支)

**注意**: 本 skill **不负责** 合并其他分支(那是合并流程,不是推送流程)。推送前应该已经把所有要合的分支都合完了, 本 skill 只做"同步 + 审查 + 推"。

## 触发词(任一即可)

- "推送"、"提交推送"、"push"、"我要推代码"
- "推上去"、"推到远程"、"提交到 GitLab/Gitee/GitHub"
- SourceTree 推送按钮点击后流程
- 用户说"我改完了, 帮我看看能不能推"

## 强制执行流程(严格顺序)

### 1. 安全前置: 确认当前分支 + 工作区干净

```bash
git branch --show-current
git status --short
```

- **必须** 工作区干净(无未提交修改)
- 如果不干净: 提示用户先 `git add` + `git commit` 或 `git stash`(参考 actp-pull 的 stash 段)
- **重要**: 记下当前分支, 推送时只允许勾选这一个分支

### 2. 二次 Fetch(推送前强制)

```bash
git fetch origin --prune
```

- 这是开工前 fetch 之后的**第二次** fetch
- 必须执行, 不能跳过——否则会错过别人在你开发期间推送的代码

### 3. 关键检查: 本地分支 vs 远程同名分支是否同步

执行 4 个值对比:

```bash
LOCAL=$(git rev-parse HEAD)
REMOTE=$(git rev-parse origin/<当前分支> 2>/dev/null)
MERGE_BASE=$(git merge-base HEAD origin/<当前分支> 2>/dev/null)

# 用 git rev-list 统计 ahead/behind
git rev-list --left-right --count HEAD...origin/<当前分支>
# 输出格式: <本地领先>    <远程领先>
```

**三种情况分别处理**:

| 情况 | ahead | behind | 处理 |
|---|---|---|---|
| **A. 完全同步** | 0 | 0 | 直接进入第 5 步(变更摘要 + 推送) |
| **B. 本地领先(无冲突)** | >0 | 0 | 直接进入第 5 步——纯快进, 无风险 |
| **C. 远程领先(本地落后)** | 0 | >0 | **必须** 先拉取到本地, 见第 4 步 |
| **D. 分叉(都有新提交)** | >0 | >0 | **必须** 先合并, 见第 4 步——这是冲突高发区 |

输出诊断:

```
【分支对齐检查】
本地:   <当前分支> @ <本地 hash 短>
远程:   origin/<当前分支> @ <远程 hash 短>
状态:   <A/B/C/D> · 本地领先 N · 远程领先 M
结论:   <可直接推送 / 需要先拉取 / 需要先合并>
```

### 4. 如果本地落后或分叉: 拉取到本地

```bash
# 先尝试快进(无本地未推送提交时最干净)
git pull --ff-only origin <当前分支>
```

- 如果 `--ff-only` 成功 → 回到第 3 步重新检查(应该变成 A 或 B)
- 如果 `--ff-only` 失败(分叉情况) → 执行 merge:
  ```bash
  git pull --no-rebase origin <当前分支>
  ```
- **禁止** 自动 `--rebase`, 团队协作会让提交历史混乱
- merge 后立刻回到第 3 步重新做对齐检查

### 5. 输出变更摘要(本 skill 的核心交付)

**这是用户最在意的输出**, 不能只说"可以推了"。

执行:

```bash
# 列出本次 push 会推上去的提交
git log --oneline origin/<当前分支>..HEAD
# 本次 push 涉及的文件统计
git diff --stat origin/<当前分支>..HEAD
# 文件级变更(可选, 详细模式)
git diff --shortstat origin/<当前分支>..HEAD
```

按以下格式输出:

```
【推送前审查】<当前分支>
推送目标: origin/<当前分支>
本地 HEAD: <hash 短>
本次推送包含 <N> 个新提交, 涉及 <M> 个文件

提交列表(从远到近):
  - <hash 短>  <类型: feat/fix/refactor/chore>  <一句话功能>
  - <hash 短>  <类型>  <说明>
  ...

改动文件分类:
  - 业务代码: <路径列表>
  - 配置文件: <路径列表, ⚠️ 需要重点 review>
  - 数据库相关: <路径列表, ⚠️ 需要同步执行 SQL>
  - 接口契约: <路径列表, ⚠️ 需要通知下游>

下一步: <如果状态 A/B: 准备推送>
        <如果状态 C/D: 已自动 pull/merge, 请确认无误后继续>
        <如果有冲突: 见下方冲突诊断段>
```

**摘要的数据源**:
1. `git log origin/<branch>..HEAD` — 列出本地未推送的提交
2. `git diff --stat origin/<branch>..HEAD` — 文件统计
3. 提交信息前缀 `feat/fix/refactor/chore`(ACTP 团队规范)
4. **项目根的 `AGENTS.md` 和 `docs/` 目录**——主动读取, 补充业务上下文(本 skill 的关键能力)
   - 例如: AGENTS.md 里写了"actp-mnzb 是直播模块", 就能告诉用户"这次推送影响了直播业务"
   - 例如: docs/ 下有 `mnzb-api.yaml`, 就能识别出哪些文件是接口契约

### 6. 无冲突场景: 推送

```bash
git push origin <当前分支>
```

推送后:

```bash
git log -1 --oneline  # 拿到最新 commit hash
```

**自动通知模板**(用户可以复制粘贴发到群):

```
【已推送】<当前分支>
分支: <当前分支>
最新提交: <hash 短>  <提交标题>
改动: <N> 个提交 / <M> 个文件
自测结果: <✅ 已通过 / ⚠️ 部分通过>
特殊变更: <数据库/配置/接口/无>
请 <小组负责人> 合并。
```

### 7. 冲突场景: 诊断 + 引导解决(本 skill 的难点)

**触发条件**: 第 4 步 merge 后产生冲突, 或第 3 步状态 D 处理后仍有冲突。

#### 7.1 收集冲突上下文

```bash
# 冲突文件列表
git diff --name-only --diff-filter=U

# 每个冲突文件的 diff(只看冲突段)
git diff --diff-filter=U

# 当前分支最近 3 个提交(看本地要推什么)
git log -3 --oneline

# 远程分支最近 3 个提交(看别人推了什么)
git log -3 --oneline origin/<当前分支>

# 冲突文件的修改历史(看谁碰过这段代码)
git log --all --oneline -- <冲突文件>
```

#### 7.2 主动读取项目业务文档

**这一步是把"git 工具"升级成"业务感知"的关键**:

```bash
# 读取项目根 AGENTS.md(团队规范和模块说明)
cat AGENTS.md 2>/dev/null | head -200

# 列出 docs/ 目录(可能含 API 文档、架构图、ADR)
ls docs/ 2>/dev/null
```

- 用 Read 工具读 `AGENTS.md`、`docs/*.md`、`docs/**/*.yaml`
- 这些文档会告诉你: 这个模块干什么、这个 API 改了什么影响哪些人、这个字段为什么这么设计

#### 7.3 输出冲突诊断报告(本 skill 的核心价值)

**绝对不要自动 `git checkout --ours` 或 `--theirs`**, 必须把决定权交给用户。

按以下格式输出:

```
【冲突诊断报告】<冲突文件>
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【冲突位置】
文件: <路径>:<行号>
冲突标记:
  <<<<<<< HEAD
  本地代码
  =======
  远程代码
  >>>>>>> origin/<当前分支>

【本地这侧的代码在做什么】
  最近提交: <hash 短>  <feat: XX功能 / fix: YY bug>
  实现意图: <根据代码 + 提交信息 + AGENTS.md 推断>
  影响范围: <这个改动影响哪些模块 / 接口 / 数据库字段>

【远程那侧的代码在做什么】
  最近提交: <hash 短>  <feat: ZZ功能 / fix: WW bug>
  实现意图: <根据代码 + 提交信息 + AGENTS.md 推断>
  影响范围: <同上>

【冲突原因分析】
  双方都修改了: <共同修改的代码段 / 配置项 / 接口签名>
  冲突类型:
    [ ] 同一行逻辑不同(两人都改了一个 if/return)
    [ ] 同名新增(两人各加了一个同名函数/字段)
    [ ] 结构性冲突(一人重构了函数, 另一人在原函数上加了逻辑)
    [ ] 配置/依赖冲突(版本号、API 路径、数据库字段)

【处理建议】
  联系 <远程提交者> 确认业务逻辑
  推荐方案:
    1. 保留双方逻辑, 用 <if/else / 参数化 / 抽象> 组合
    2. 选本地版本(适用: <条件>)
    3. 选远程版本(适用: <条件>)

  警告: 禁止擅自选一侧; 禁止为消除冲突而删除他人有效代码
```

**诊断的判断逻辑**:
1. 冲突段前后 10 行代码 → 推断这段代码的"用途"
2. `git log -1` 双方最近提交信息 → 推断"功能意图"
3. 读 `AGENTS.md` 了解模块职责 → 推断"业务影响"
4. 读 `docs/` 相关文档 → 推断"接口契约"
5. 综合上面 4 步, 用自然语言告诉用户"两边分别实现什么, 为什么撞车"

#### 7.4 引导用户解决

```bash
# 用户编辑文件后, 标记为已解决
git add <已解决的冲突文件>
git commit -m "merge: 解决 <冲突文件> 与 origin/<当前分支> 的冲突"
```

- **绝对不要** 自动 `--ours` / `--theirs` / `-X theirs`
- 用户在 SourceTree 看不到"中止合并"时: `git merge --abort` 可回到合并前状态

### 8. 推送后通知

不管是否经过冲突解决, 推送成功后:

```bash
git log -1 --oneline
```

输出"已推送"模板(同第 6 步末尾), 用户发到群里通知负责人。

## 禁止事项(踩坑警示)

- 禁止跳过第 2 步 fetch(错过别人推送, 直接 push 会失败或覆盖)
- 禁止跳过第 3 步对齐检查(直接 push 可能推到错误分支)
- 禁止 `git push -f` / `--force`(强制覆盖团队分支, 灾难性)
- 禁止勾选"所有分支"推送(普通成员只许推自己分支)
- 禁止在有未提交修改时 push
- 禁止自动解决冲突(`--ours` / `--theirs` / `git add .` 后直接 commit)
- 禁止跳过第 5 步变更摘要(失去本 skill 核心价值)
- 禁止跳过第 7 步业务上下文读取(只凭代码 diff 推断意图, 容易误判冲突原因)
